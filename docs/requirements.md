# 平和島ボートレース予想アプリ 要件定義書

## 1. プロジェクト概要

### 1.1 目的

ボートレース平和島に特化した舟券購入支援アプリを開発する。当日のレース情報を自動取得し、ユーザーが入力した掛け金の範囲内で期待値の高い買い目を提案する。

### 1.2 対象レース場

- **平和島競艇場**（場コード: `04`）
- 所在地: 東京都大田区平和島1丁目1-1

### 1.3 技術スタック

| 項目 | 技術 |
|---|---|
| 言語 | Python 3.11+ |
| Webフレームワーク | Streamlit（プロトタイプ） → FastAPI + React（本番） |
| データ取得 | requests + BeautifulSoup4 / `metaboatrace.scrapers` |
| 機械学習 | LightGBM（メインモデル）、scikit-learn |
| データベース | SQLite（開発）→ PostgreSQL（本番） |
| スケジューラ | APScheduler / cron |

---

## 2. 平和島コースの特性（予測モデルに反映すべき要素）

予測精度を上げるために、平和島固有の水面特性をモデルに組み込む必要がある。

| 特性 | 詳細 | 予測への影響 |
|---|---|---|
| 1マーク〜スタンド幅 | **37m**（全国最狭） | インコースの逃げが効きにくい。1コース1着率は約44.9%（全国平均55.3%） |
| 差しの決まりやすさ | バックストレッチ斜行禁止規則あり | 2〜4コースの差し展開が頻発。出足・行き足の良い艇が有利 |
| 水質 | 海水（東京湾からの潮流） | 満潮時は1マークにうねり発生 → インスロー有利。干潮時はダッシュ勢有利 |
| 風 | 春〜夏：追い風 / 冬：向かい風 | 追い風 → イン有利 / 向かい風 → アウト有利 |
| 5・6コース1着率 | 全国平均より高い | 万舟出現率が高い。高配当狙いが可能 |
| モーター交換時期 | 例年6月頃 | 交換直後（6〜8月）は展示タイム重視 |

## 3. 機能要件

### 3.1 データ自動取得機能

#### 3.1.1 取得対象データ

公式サイト（`boatrace.jp`）から以下のデータをスクレイピングで取得する。

**出走表データ（レース前）**

| データ項目 | URL パターン | 備考 |
|---|---|---|
| 出走表 | `/owpc/pc/race/racelist?hd={YYYYMMDD}&jcd=04&rno={1-12}` | 選手情報・枠番・モーター/ボート番号 |
| オッズ（3連単） | `/owpc/pc/race/odds3t?hd={YYYYMMDD}&jcd=04&rno={1-12}` | 買い目選定に使用 |
| オッズ（3連複） | `/owpc/pc/race/odds3f?hd={YYYYMMDD}&jcd=04&rno={1-12}` | 買い目選定に使用 |
| オッズ（2連単） | `/owpc/pc/race/odds2tf?hd={YYYYMMDD}&jcd=04&rno={1-12}` | 買い目選定に使用 |
| 直前情報 | `/owpc/pc/race/beforeinfo?hd={YYYYMMDD}&jcd=04&rno={1-12}` | 展示タイム・水面気象情報 |

**選手・機材データ（蓄積用）**

- 選手別成績（全国勝率・当地勝率・コース別成績）
- モーター2連率・ボート2連率
- 今節成績（節間の着順推移）

**水面気象データ**

- 風向・風速
- 波高
- 気温・水温
- 潮位（満潮/干潮時刻）

#### 3.1.2 取得タイミング

| タイミング | 取得内容 |
|---|---|
| 毎朝（開催日） | 出走表・選手データ・モーターデータ |
| レース直前（発走15分前） | 直前情報（展示タイム・水面気象）・最新オッズ |
| レース後 | レース結果・払戻金（学習データ蓄積用） |

#### 3.1.3 データソース

| 優先度 | ソース | 説明 |
|---|---|---|
| 1 | [BOAT RACE 公式](https://www.boatrace.jp/) | 出走表・オッズ・直前情報・結果 |
| 2 | [公式データダウンロード](https://www.boatrace.jp/owpc/pc/extra/data/download.html) | 過去の成績データ（LZH形式） |
| 3 | [metaboatrace/scrapers](https://github.com/metaboatrace/scrapers) | Python用スクレイピングライブラリ |
| 4 | [艇国データバンク](https://boatrace-db.net/) | コース別成績の参考データ |

### 3.2 予測エンジン

#### 3.2.1 予測対象

各レースにおける **6艇の着順確率分布** を推定し、それを基に全買い目（3連単: 120通り / 3連複: 20通り / 2連単: 30通り / 2連複: 15通り）の的中確率を算出する。

#### 3.2.2 特徴量設計

予測モデルに入力する特徴量を以下のカテゴリに分類する。

**A. 選手特徴量**

| 特徴量 | 説明 |
|---|---|
| 全国勝率 | 直近6ヶ月の全国成績 |
| 当地勝率 | 平和島での過去3年間の成績 |
| コース別1着率 | 各コース（1〜6）での1着率 |
| 平均ST（スタートタイミング） | コース別の平均スタートタイミング |
| フライング/出遅れ回数 | 直近のF/L回数（F持ちは逃げにくい） |
| 級別 | A1 / A2 / B1 / B2 |
| 今節成績 | 当節内の着順推移 |

**B. 機材特徴量**

| 特徴量 | 説明 |
|---|---|
| モーター2連率 | モーターの直近の2連率 |
| ボート2連率 | ボートの直近の2連率 |
| モーター使用期間 | 交換後の経過月数（新モーター期はデータ信頼度低） |

**C. 平和島固有特徴量**

| 特徴量 | 説明 |
|---|---|
| 潮位 | レース時の潮位（満潮/干潮/中間） |
| 潮位変動方向 | 上げ潮 / 下げ潮 |
| 風向 | 追い風 / 向かい風 / 横風 |
| 風速 | m/s |
| 波高 | cm |
| 季節フラグ | 春夏秋冬（季節による傾向差をエンコード） |

**D. レースコンテキスト特徴量**

| 特徴量 | 説明 |
|---|---|
| レースグレード | SG / G1 / G2 / G3 / 一般 |
| レース番号 | 第1R〜第12R（番号による傾向差） |
| 枠番 | 1〜6号艇 |
| 進入コース予測 | 枠なり / 前付け傾向の有無 |

**E. 直前特徴量（レース直前に取得）**

| 特徴量 | 説明 |
|---|---|
| 展示タイム | 周回展示のタイム |
| チルト角度 | モーターのチルト角 |
| 部品交換情報 | エンジン部品の交換有無と種類 |

#### 3.2.3 モデル構成

```
┌──────────────────────────────────────────────────┐
│                予測パイプライン                     │
│                                                    │
│  ┌────────────┐    ┌─────────────┐    ┌─────────┐ │
│  │ 特徴量生成  │───▶│ LightGBM    │───▶│ 着順確率 │ │
│  │ (前処理)    │    │ ランキング  │    │ 分布    │ │
│  └────────────┘    │ 学習モデル  │    └────┬────┘ │
│                     └─────────────┘         │      │
│                                             ▼      │
│                     ┌─────────────┐    ┌─────────┐ │
│                     │ オッズ情報   │───▶│ 期待値   │ │
│                     │             │    │ 算出    │ │
│                     └─────────────┘    └─────────┘ │
└──────────────────────────────────────────────────┘
```

- **アルゴリズム**: LightGBM（LambdaRank）をメインモデルとして採用
  - 理由: ランキング学習は着順予測に適しており、特徴量の重要度が解釈しやすい
- **学習データ**: 平和島のみに絞った過去3年分のレースデータ（約15,000レース）
- **再学習頻度**: 週次で最新データを追加して再学習

#### 3.2.4 期待値算出ロジック

```
期待値 = 予測的中確率 × オッズ × 100円（1点あたり）

期待値 > 1.0 の買い目 = プラス期待値（推奨買い目候補）
```

### 3.3 買い目提案機能

#### 3.3.1 入力

| 項目 | 型 | 説明 |
|---|---|---|
| 掛け金（予算） | 整数（円） | レース単位 or 全レース通しで指定 |
| 舟券種別 | 選択式 | 3連単 / 3連複 / 2連単 / 2連複 / ワイド |
| 戦略モード | 選択式 | 下記参照 |

#### 3.3.2 戦略モード

| モード | 説明 | 対象ユーザー |
|---|---|---|
| **回収率重視** | 期待値が最も高い買い目を選定。点数を絞り込む | 堅実派 |
| **的中率重視** | 的中確率が高い買い目を広めにカバー | 的中体験重視派 |
| **万舟狙い** | 高配当（オッズ50倍以上）かつ一定の的中確率がある買い目を選定 | 一発狙い派 |
| **バランス** | 期待値と的中率のバランスを加重平均で最適化 | デフォルト |

#### 3.3.4 資金配分アルゴリズム

**ケリー基準（修正版）** を採用し、各買い目への最適な資金配分を算出する。

```
f* = (p × b - q) / b

f*: 資金の最適配分比率
p : 予測的中確率
b : オッズ - 1（純利益倍率）
q : 1 - p（外れる確率）
```

- ケリー基準の結果をそのまま使うとリスクが高いため、**ハーフケリー**（f* / 2）を推奨デフォルトとする
- 最低賭け金は100円単位に丸める

## 4. 精度向上のためのアイデア・提案

### 4.1 モデリング面

| # | アイデア | 効果 | 実装難易度 |
|---|---|---|---|
| 1 | **平和島データのみで学習** | 全場混合より場固有の傾向を捉えやすい。1コース逃げ率の低さや差し展開の多さを正確に反映 | 低 |
| 2 | **LightGBMランキング学習（LambdaRank）** | 二値分類（1着か否か）よりも着順の相対関係を学習でき、予測精度が向上 | 中 |
| 3 | **アンサンブル学習** | LightGBM + ニューラルネット + ロジスティック回帰の予測を加重平均し、個別モデルの弱点を補完 | 中 |
| 4 | **Borutaによる特徴量選択** | 不要な特徴量を除去し、過学習を防止。モデルの汎化性能向上 | 低 |
| 5 | **定期再学習パイプライン** | 週次で最新データを追加学習。選手・モーターのコンディション変化に追従 | 中 |
| 6 | **モデルキャリブレーション** | Platt Scaling等で予測確率を校正し、期待値算出の精度を向上 | 低 |

### 4.2 特徴量エンジニアリング面

| # | アイデア | 効果 | 実装難易度 |
|---|---|---|---|
| 7 | **潮位×風向の交差特徴量** | 平和島では潮位と風向の組み合わせがレース展開に大きく影響。交差項で非線形な影響を捕捉 | 低 |
| 8 | **選手×コース相性スコア** | 各選手のコース別成績（特に平和島での実績）をスコア化。平和島巧者を検出 | 中 |
| 9 | **モーター上昇/下降トレンド** | モーター2連率の推移（直近5節の傾向）。整備力のある選手がモーターを仕上げていく過程を捕捉 | 中 |
| 10 | **スタートタイミング安定度** | 平均STだけでなく標準偏差も特徴量化。安定したスタートができる選手を高評価 | 低 |
| 11 | **前走レース間隔** | 同日の前レースとの間隔。連続出走の疲労影響を考慮 | 低 |
| 12 | **展示タイム偏差** | 展示タイムの絶対値ではなく、当レース内での相対的な速さ（偏差値化） | 低 |

### 4.3 オッズ活用面

| # | アイデア | 効果 | 実装難易度 |
|---|---|---|---|
| 13 | **オッズの歪み検出** | モデル予測確率とオッズ暗示確率の乖離を検出し、期待値の高い買い目を自動抽出 | 中 |
| 14 | **締め切り直前オッズ取得** | オッズは締め切り直前が最も正確。直前取得で期待値計算の精度向上 | 中 |
| 15 | **オッズ変動パターン分析** | 特定の買い目に大量投票が入るパターン（プロの動き）を検出 | 高 |

### 4.4 運用・評価面

| # | アイデア | 効果 | 実装難易度 |
|---|---|---|---|
| 16 | **バックテスト基盤** | 過去データでのシミュレーションにより戦略検証。過学習の防止にも有効 | 中 |
| 17 | **的中率/回収率ダッシュボード** | モデル性能を常時監視し、精度劣化を早期検知 | 低 |

## 5. 非機能要件

### 5.1 性能

| 項目 | 要件 |
|---|---|
| 予測応答時間 | 1レースあたり3秒以内 |
| データ取得間隔 | 公式サイトへの負荷を考慮し、リクエスト間隔1秒以上 |
| 同時利用者数 | 個人利用想定（1ユーザー） |

### 5.2 データ管理

| 項目 | 要件 |
|---|---|
| データ保存期間 | 過去3年分のレースデータを保持 |
| バックアップ | 日次自動バックアップ |
| データ形式 | CSV（生データ）+ DB（加工済みデータ） |

### 5.3 法的・倫理的配慮

| 項目 | 対応方針 |
|---|---|
| スクレイピング | robots.txt を遵守。サーバー負荷を最小限にする（1秒以上の間隔） |
| 利用規約 | boatrace.jp の利用規約を確認し、個人利用の範囲内で使用 |
| 免責事項 | アプリ内に「予測は参考情報であり、舟券購入は自己責任」の明示 |
| ギャンブル依存対策 | 1日の投資上限設定機能を実装 |

## 7. ディレクトリ構成（案）

```
heiwajima-predictor/
├── README.md
├── requirements.txt
├── config/
│   └── settings.yaml          # 設定ファイル（場コード、取得間隔等）
├── src/
│   ├── scraper/
│   │   ├── race_list.py       # 出走表取得
│   │   ├── odds.py            # オッズ取得
│   │   ├── before_info.py     # 直前情報取得
│   │   ├── results.py         # レース結果取得
│   │   └── weather.py         # 水面気象データ取得
│   ├── database/
│   │   ├── models.py          # ORMモデル定義
│   │   ├── migrations/        # マイグレーション
│   │   └── repository.py      # データアクセス層
│   ├── features/
│   │   ├── player.py          # 選手特徴量
│   │   ├── machine.py         # 機材特徴量
│   │   ├── course.py          # 平和島固有特徴量
│   │   └── pipeline.py        # 特徴量生成パイプライン
│   ├── model/
│   │   ├── trainer.py         # モデル学習
│   │   ├── predictor.py       # 予測実行
│   │   ├── evaluator.py       # モデル評価
│   │   └── saved_models/      # 学習済みモデル保存
│   ├── betting/
│   │   ├── expected_value.py  # 期待値算出
│   │   ├── kelly.py           # ケリー基準ベース資金配分
│   │   └── strategy.py        # 戦略モード制御
│   └── app/
│       ├── streamlit_app.py   # Streamlit UI
│       ├── components/        # UIコンポーネント
│       └── pages/             # 各画面
├── data/
│   ├── raw/                   # 生データ（CSV）
│   ├── processed/             # 加工済みデータ
│   └── heiwajima.db           # SQLiteデータベース
├── notebooks/
│   ├── eda.ipynb              # 探索的データ分析
│   └── model_experiment.ipynb # モデル実験
└── tests/
    ├── test_scraper.py
    ├── test_features.py
    ├── test_model.py
    └── test_betting.py
```
